New architecture

Ron as a Bun workspace with modular API client packages

Throw away the old single-app shape. The new system is a Bun monorepo where every external provider is wrapped in its own reusable client package, service packages compose those clients, and the Hono app only orchestrates domain workflows and HTTP delivery.

Runtime

Bun workspaces

Fast local dev, one lockfile, package-level scripts

Boundary

API client packages

One package per external API with typed requests, retries, and errors

App layer

Thin Hono app

Routes call services; services call packages; clients hit providers

Workspace shape

The repo becomes package-first. Shared contracts and helpers sit at the bottom, provider clients stay isolated, and application code lives in apps/.

Design rules

1. One provider, one package

Each external API gets its own Bun package with typed endpoints, auth/config, retry behavior, and provider-specific errors.

2. Normalize above the clients

Client packages should expose provider-native shapes. Service packages map those shapes into shared Ron domain contracts.

3. No provider logic in routes

The Hono app imports service packages only. It should never import Weather.gov, FEMA, USGS, NASA, or geo lookup clients directly.

4. Shared transport stays shared

Timeouts, retries, rate limiting, headers, and tracing belong in packages/http so every client behaves consistently.

These provider packages are the new hard boundaries in the system. Each should be publishable, mockable, and independently testable.

HTTP API surface

The Hono app exposes Ron domain data only — never raw provider payloads. All responses include a generatedAt timestamp. Errors follow RFC 9457 Problem Details.

Method Path Query params Returns
GET /health — Liveness status for the API process itself
GET /openapi.json — OpenAPI 3.1 contract generated from packages/contract
GET /disasters source category severity status state limit DisasterSearchResponse — paginated list of normalized events
GET /disasters/:id — Single DisasterEvent
GET /disasters/snapshot — DisasterSnapshot — full assembled view for dashboards
GET /zip-codes/:zip/impact source ZipImpactResponse — resolved location + matching disasters

Client package contract

Every provider package should feel the same to consume even when the provider APIs behave differently.

Typed constructor

Accept base URL overrides, headers, timeout, retries, and user-agent information through a typed config object.

Endpoint modules

Split endpoints into files like alerts.ts, events.ts, or geographies.ts so packages stay modular internally too.

Provider-native types

Do not leak Ron normalization into the packages. Keep external schemas intact and map them in service packages.

Explicit errors

Surface rate limits, not-found cases, invalid provider payloads, and upstream failures with distinct error types.

Recommended implementation order

Phase 1

Create Bun workspaces, shared contract, geo, and http packages

Phase 2

Implement one package per external API with fixtures and package-level tests

Phase 3

Build service packages that normalize provider-native payloads into Ron contracts

Phase 4

Wire the Hono app to services only and generate API docs from shared contract models

Phase 5

Attach the demo UI and add end-to-end flows across app, services, and provider packages

Testing strategy

Unit tests live inside each package and never hit the network. Integration tests cross package boundaries and live at the workspace root. The rule: if it can be faked with a fixture, it must be.

Provider client units

Each api-* package ships fixtures alongside its tests. Mock HTTP at the fetch level so no live network is needed.

Service integrations

Service-level tests cross multiple packages. They verify that normalization, matching, and geo logic produce correct Ron domain output end-to-end.

Live parity flag

Integration tests can optionally run against real providers with TEST_LIVE=true. Gated from CI by default to avoid flakiness.

API design rules

Stable envelopes

Every collection response wraps items in { generatedAt, count, items }. Clients should never parse bare arrays.

String enums

All enum fields serialize as explicit strings ("flood", "severe") — never integers or computed values.

No provider leakage

Response fields come from the Ron contract. Provider-native IDs, field names, and quirks are normalized away before they reach the route layer.

RFC 9457 errors

Error responses follow Problem Details (type, title, status, detail) so clients can handle failures consistently.
